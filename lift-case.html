Lift 2.0 — Escalando Design Systems para múltiplas marcas

Como estruturamos um sistema capaz de suportar diferentes identidades sem perder consistência e velocidade.

Contexto

Um grupo educacional em expansão digital acelerada

A YDUQS é um dos maiores grupos educacionais do Brasil. Sob seu guarda-chuva operam marcas com identidades e públicos distintos — IBMEC, Estácio, entre outras — cada uma com suas plataformas digitais, portais do aluno e ferramentas internas.

Com a aceleração digital pós-pandemia, o número de produtos cresceu rapidamente. Mas a infraestrutura de design não acompanhou: cada time criava seus próprios componentes, definia suas próprias cores, e tomava decisões visuais de forma isolada. O resultado era previsível — inconsistência, retrabalho e lentidão.

Desafio

O problema não era visual — era organizacional

O Lift 1.0 existia como um conjunto de UI Kits no Figma, mas não funcionava como um sistema de verdade. Não havia tokens compartilhados, não havia governança, e não havia alinhamento entre o que o design definia e o que o front-end implementava.

O desafio real ia além de componentes bonitos:

Múltiplas marcas, uma estrutura

IBMEC e Estácio têm identidades visuais completamente diferentes, mas precisam compartilhar a mesma base de componentes para viabilizar escala.

Desalinhamento entre times

Designers tomavam decisões visuais sem referência comum. Desenvolvedores implementavam variações que divergiam do Figma. Ninguém sabia qual era a "versão correta".

Limitações técnicas

Os componentes existentes eram acoplados a contextos específicos — impossível reutilizar um botão do IBMEC na Estácio sem reescrever código.

Padronizar sem engessar

O sistema precisava garantir consistência sem tirar a autonomia dos times. Regras demais matam velocidade; regras de menos geram caos.

Problema

Cada falha tinha uma consequência real

  • Tokens pouco estruturados geravam retrabalho constante — cada novo projeto reinventava cores e espaçamentos do zero.
  • Baixa padronização entre marcas fazia com que a mesma feature tivesse aparências diferentes em produtos do mesmo grupo.
  • Componentes acoplados impediam reutilização — um formulário criado para o IBMEC não servia para a Estácio.
  • Ausência de governança significava que ninguém sabia quem mantinha o quê, gerando duplicação e decisões conflitantes.
  • Gap entre design e código criava um ciclo de revisões infinitas — o Figma dizia uma coisa, o front-end entregava outra.
Decisões chave

Três apostas que definiram o projeto

01

Separar tokens de brand e usage

O que: Criar camadas distintas para identidade visual (brand) e aplicação (semantic/usage).

Por quê: Permitir que o mesmo componente sirva múltiplas marcas sem duplicação. A troca de tema se torna uma troca de variáveis, não de componentes.

02

Criar camadas de abstração

O que: Arquitetura em 4 níveis — Base → Brand → Semantic → Component.

Por quê: Desacoplar identidade de aplicação. Cada camada tem uma responsabilidade clara, reduzindo a complexidade de manutenção.

03

Priorizar governança antes de expansão

O que: Estabelecer processos de contribuição, revisão e versionamento antes de escalar o número de componentes.

Por quê: Um sistema sem governança escala problemas, não soluções. Precisávamos de regras claras antes de crescer.

Trade-offs

O que aceitamos para chegar lá

Nenhuma decisão de design system é gratuita. Cada escolha envolve um custo — e reconhecer isso é parte da maturidade do processo.

Aumentamos a complexidade inicial da arquitetura de tokens
→
Para ganhar escalabilidade real a longo prazo
Priorizamos consistência mesmo com impacto inicial na velocidade
→
Para eliminar retrabalho recorrente entre times
Investimos tempo em governança antes de criar novos componentes
→
Para garantir que o crescimento fosse sustentável
Arquitetura

Quatro camadas, uma responsabilidade cada

A separação em camadas é o que permite trocar a identidade visual de uma marca inteira sem quebrar um único componente.

01 — Base

Primitivos

Valores absolutos: paleta de cores, escala tipográfica, grid de espaçamento. Base imutável do sistema.

↓
02 — Brand

Identidade

Mapeamento de primitivos para cada marca. IBMEC usa um set, Estácio usa outro — mesma estrutura, valores diferentes.

↓
03 — Semantic

Intenção

Tokens de uso contextual: background-surface, text-primary, border-subtle. Mapeiam intenção, não valor.

↓
04 — Component

Interface

Componentes reutilizáveis que consomem tokens semânticos. Agnósticos de marca — um botão serve todos os produtos.

Execução

O que foi construído

Token Structure

Hierarquia de tokens em 4 camadas com variáveis no Figma sincronizadas com CSS custom properties. Cada marca tem seu próprio set de brand tokens.

Component System

Biblioteca de componentes com estados completos, variantes documentadas e API consistente. Construídos para consumir tokens semânticos, não valores fixos.

Multi-brand Engine

Suporte nativo a múltiplas marcas via brand tokens. Trocar o tema de IBMEC para Estácio é trocar um arquivo de variáveis — zero alteração em componentes.

Living Documentation

Documentação viva com guidelines de uso, padrões de contribuição, changelog versionado e exemplos interativos para designers e desenvolvedores.

Impacto no time

O que mudou na forma de trabalhar

Designers mais autônomos

Com tokens e componentes bem definidos, designers passaram a tomar decisões com confiança — sem precisar validar cada escolha visual com liderança.

Menos decisões manuais

O sistema absorveu decisões repetitivas (cores, espaçamentos, estados). O time passou a focar em problemas de produto, não em detalhes de UI.

Previsibilidade na entrega

Com componentes prontos e documentados, a estimativa de esforço ficou mais precisa. Menos surpresas no handoff design → desenvolvimento.

Alinhamento design ↔ código

Variáveis do Figma mapeadas 1:1 com CSS custom properties. O que o designer vê é o que o desenvolvedor implementa — sem tradução manual.

Antes vs Depois

A transformação

Antes — Lift 1.0
  • UI Kits isolados por marca
  • Tokens não documentados
  • Componentes duplicados e acoplados
  • Governança informal
  • Gap constante entre design e código
  • Cada projeto reinventava padrões
Depois — Lift 2.0
  • Sistema unificado multi-marca
  • Tokens em 4 camadas documentadas
  • Componentes agnósticos e reutilizáveis
  • Governança com processos claros
  • Variáveis Figma ↔ CSS sincronizadas
  • Novos produtos herdam o sistema
Aprendizados

O que esse projeto me ensinou

  • Design System é um problema organizacional, não apenas visual. A parte mais difícil não é criar componentes — é mudar como times trabalham juntos.
  • Governança é o que separa um kit de UI de um sistema de verdade. Sem processos claros de contribuição e revisão, o sistema se degrada com o tempo.
  • Separação de camadas reduz complexidade a longo prazo. O investimento inicial em arquitetura se paga exponencialmente quando novas marcas entram no ecossistema.
  • Documentação acessível acelera adoção mais do que treinamentos formais. Times adotam o que conseguem entender sozinhos.
  • Design System é um produto — precisa de roadmap, métricas e iteração contínua. Não é um projeto com data de fim.
