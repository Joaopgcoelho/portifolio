CubiCare — Simplificando a rotina alimentar infantil

Como criei um produto para organizar refeições de crianças e transformar um processo complexo em algo simples e previsível.

Contexto

O desafio da alimentação infantil

Quando um bebê começa a introdução alimentar, os pais enfrentam uma rotina exaustiva: preparar refeições variadas todos os dias, garantir equilíbrio nutricional entre grupos alimentares, e manter controle do que está disponível no freezer.

A técnica de batch cooking com cubos congelados resolve parte do problema — você cozinha em lote e congela em porções de 25ml. Mas organizar mentalmente o que tem pronto, o que está acabando, e o que precisa produzir continua sendo uma carga cognitiva alta.

Problema

Rotina imprevisível e esforço constante

  • Pais não sabem quantas refeições completas têm prontas
  • Descobrem que falta algo na hora de montar o prato
  • Não há visibilidade sobre o equilíbrio entre grupos nutricionais
  • O planejamento de produção é mental — fácil de esquecer
  • Cada refeição exige decisão: "o que combinar hoje?"

"O problema não era cozinhar — era organizar. Os pais já sabiam preparar os alimentos. O que faltava era visibilidade e previsibilidade sobre o que estava disponível."

Concepção

A ideia: cubos como unidade de valor

O insight central foi tratar cada cubo congelado como uma unidade atômica do sistema. Se você tem cubos suficientes de cada grupo nutricional, tem refeições completas. Se algum grupo acaba, o sistema avisa.

A partir disso, defini a estrutura do produto:

  • 5 grupos nutricionais — proteína, leguminosa, vegetal A, vegetal B, carboidrato
  • Estoque em cubos — cada grupo tem uma meta (ex: 30 cubos)
  • Refeição = 1 cubo de cada grupo — montagem automática
  • Ciclo claro — Comprar → Produzir → Estocar → Montar Refeição
Solução

App mobile-first para controle alimentar

Criei um app PWA com React + TypeScript focado em resolver o problema com o mínimo de fricção possível. As decisões de produto foram:

  • Mobile-first — usado na cozinha, com bebê no colo
  • Estoque como dashboard — a home responde "quantas refeições tenho?"
  • Produção guiada — o app diz o que produzir, não o contrário
  • Registro em 1 toque — sugestão de refeição + botão de confirmar
  • Lanches separados — café e lanche são itens únicos (fruta, iogurte), não refeições completas
Experiência

O fluxo principal

O produto foi estruturado em torno de um ciclo que se repete semanalmente:

Comprar → Produzir (batch cooking) → Estocar cubos → Montar refeições → Repetir

Cada etapa do ciclo tem uma tela dedicada no app. O Dashboard conecta tudo mostrando o estado atual e a próxima ação necessária.

Decisões

Escolhas de produto

Mobile-first Usado na cozinha, com uma mão livre
Ciclos Produção → Consumo como modelo mental
Refeições Estoque traduzido em valor real

A decisão mais importante foi transformar "cubos em estoque" em "refeições disponíveis". O pai não pensa em cubos — pensa em "quantos dias estou coberto". Essa tradução é o core do produto.

Primeira versão

MVP funcional

A primeira versão entregou as funcionalidades essenciais:

  • Dashboard com contagem de refeições e status por grupo
  • Estoque com adição/remoção de cubos por grupo
  • Registro de refeições com desconto automático do estoque
  • Sugestões de produção baseadas no que está baixo
  • Lista de compras gerada automaticamente

O produto funcionava. Mas usar no dia a dia revelou problemas que não eram visíveis no design.

Aprendizados

O que o uso real revelou

  • O conceito não era óbvio — novos usuários não entendiam "cubos" sem explicação
  • Excesso de informação na home — 6 barras de progresso + alertas + frutas = ruído
  • Falta de direção — o app mostrava dados mas não dizia o que fazer
  • Registro de refeição era pesado — 6 selects em um modal para montar um prato
  • Frutas tinham lógica separada — um calendário semanal que não fazia sentido no fluxo
  • Visual corporativo — azul frio e glass morphism não combinavam com "cuidar do bebê"
Evolução

Refinamento baseado em uso

Com base nos aprendizados, fiz uma evolução completa do produto mantendo a lógica de negócio intacta:

  • Onboarding — 3 telas que explicam o conceito de cubos antes de usar
  • Dashboard contextual — muda baseado no horário (café, almoço, lanche, jantar)
  • One-tap meal — registrar refeição com 1 toque direto da home
  • Quick Snack — bottom sheet para registrar café/lanche (banana, iogurte, panquequinha)
  • Status emocional — "🟢 Semana tranquila" ao invés de 6 barras de progresso
  • Paleta quente — verde-sage + creme, emojis nos grupos, tom acolhedor
  • Navegação simplificada — 6 itens → 5, com FAB para ações rápidas
Transformação

Antes e depois

Antes Experiência confusa, alta carga cognitiva, sem direção clara
Depois Fluxo guiado, decisões claras, experiência leve

"O app deixou de mostrar dados e passou a guiar ações. A pergunta mudou de 'o que esses números significam?' para 'ok, o que faço agora?' — e a resposta está sempre visível."

Sistema

Design System enxuto

Reduzi de 49 componentes UI para 12. Padronizei cores, espaçamentos e border-radius. Criei componentes reutilizáveis (EmptyState, PageHeader, QuickSnack) que garantem consistência sem overhead.

  • Paleta semântica: primary (sage), warning (amber), danger (warm red)
  • Tipografia: Inter, escala de 11px a 34px
  • Espaçamento: sistema de 4px (4, 8, 12, 16, 24, 32)
  • Componentes: Button, Card (.glass), Progress, AlertBanner, EmptyState, Dialog
  • Emojis como identidade visual dos grupos nutricionais
Impacto

Resultado

1 toque Para registrar uma refeição completa
-38% CSS bundle (138KB → 86KB)
12 Componentes (eram 49)

O produto se tornou mais compreensível, mais rápido de usar, e mais agradável visualmente. A experiência passou de "dashboard de dados" para "assistente de rotina".

Reflexões

O que aprendi

  • Criar produto é diferente de torná-lo utilizável — funcionar não é suficiente, precisa ser claro
  • Clareza é a feature mais importante — se o usuário não entende em 5 segundos, o design falhou
  • UX é direcionar comportamento — não mostrar dados, mas guiar a próxima ação
  • Contexto temporal importa — o que o app mostra às 7h deve ser diferente das 19h
  • Evolução contínua faz parte do produto — a v1 é hipótese, o uso real é validação
