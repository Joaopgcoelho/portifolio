Lift 2.0 — Scaling Design Systems for multiple brands

How we structured a system capable of supporting different identities without losing consistency and speed.

↓ 70% Inconsistency across products
3x UI delivery speed
90% DS adoption across teams
90% CSAT on SAVA (first product)
Design System System Ops Governance Multi-brand Figma
Context

An educational group in accelerated digital expansion

YDUQS is one of the largest educational groups in Brazil. Under its umbrella operate brands with distinct identities and audiences — IBMEC, Estácio, among others — each with their own digital platforms, student portals, and internal tools.

With the post-pandemic digital acceleration, the number of products grew rapidly. But the design infrastructure didn't keep up: each team created their own components, defined their own colors, and made visual decisions in isolation. The result was predictable — inconsistency, rework, and slowness.

Challenge

The problem wasn't visual — it was organizational

Lift 1.0 existed as a set of UI Kits in Figma, but it didn't function as a real system. There were no shared tokens, no governance, and no alignment between what design defined and what the front-end implemented.

The real challenge went beyond beautiful components:

Multiple brands, one structure

IBMEC and Estácio have completely different visual identities, but they need to share the same component base to enable scale.

Misalignment between teams

Designers made visual decisions without a common reference. Developers implemented variations that diverged from Figma. Nobody knew which was the "correct version."

Technical limitations

Existing components were coupled to specific contexts — impossible to reuse an IBMEC button in Estácio without rewriting code.

Standardize without constraining

The system needed to ensure consistency without removing team autonomy. Too many rules kill speed; too few rules create chaos.

Problem

Every failure had a real consequence

  • Poorly structured tokens generated constant rework — every new project reinvented colors and spacing from scratch.
  • Low standardization across brands meant the same feature looked different across products within the same group.
  • Coupled components prevented reuse — a form built for IBMEC didn't work for Estácio.
  • Lack of governance meant nobody knew who maintained what, leading to duplication and conflicting decisions.
  • Gap between design and code created an endless revision cycle — Figma said one thing, the front-end delivered another.
Key decisions

Three bets that defined the project

01

Separate brand and usage tokens

What: Create distinct layers for visual identity (brand) and application (semantic/usage).

Why: Allow the same component to serve multiple brands without duplication. Theme switching becomes a variable swap, not a component swap.

02

Create abstraction layers

What: Architecture in 4 levels — Core → Brand → Semantic → Component.

Why: Decouple identity from application. Each layer has a clear responsibility, reducing maintenance complexity.

03

Prioritize governance before expansion

What: Establish contribution, review, and versioning processes before scaling the number of components.

Why: A system without governance scales problems, not solutions. We needed clear rules before growing.

Trade-offs

What we accepted to get there

No design system decision is free. Every choice involves a cost — and recognizing that is part of the process maturity.

Architecture

Four layers, one responsibility each

The layer separation is what allows swapping an entire brand's visual identity without breaking a single component.

01 — Core

Primitives

Absolute values: color palette, typographic scale, spacing grid. The system's immutable foundation.

↓
02 — Brand

Identity

Mapping primitives to each brand. IBMEC uses one set, Estácio uses another — same structure, different values.

↓
03 — Semantic

Intent

Contextual usage tokens: background-surface, text-primary, border-subtle. They map intent, not value.

↓
04 — Component

Interface

Reusable components that consume semantic tokens. Brand-agnostic — one button serves all products.

Execution

What was built

Token Structure

4-layer token hierarchy with Figma variables synced to CSS custom properties. Each brand has its own set of brand tokens.

Component System

Component library with complete states, documented variants, and a consistent API. Built to consume semantic tokens, not fixed values.

Multi-brand Engine

Native multi-brand support via brand tokens. Switching the theme from IBMEC to Estácio means swapping a variables file — zero component changes.

Living Documentation

Living documentation with usage guidelines, contribution patterns, versioned changelog, and interactive examples for designers and developers.

Impact

Team impact

What changed in how we work

More autonomous designers

With well-defined tokens and components, designers started making decisions with confidence — without needing to validate every visual choice with leadership.

Fewer manual decisions

The system absorbed repetitive decisions (colors, spacing, states). The team shifted focus to product problems, not UI details.

Delivery predictability

With ready and documented components, effort estimation became more accurate. Fewer surprises in the design → development handoff.

Design ↔ code alignment

Figma variables mapped 1:1 with CSS custom properties. What the designer sees is what the developer implements — no manual translation.

Key takeaways

What this project taught me

  • A Design System is an organizational problem, not just a visual one. The hardest part isn't creating components — it's changing how teams work together.
  • Governance is what separates a UI kit from a real system. Without clear contribution and review processes, the system degrades over time.
  • Layer separation reduces long-term complexity. The initial investment in architecture pays off exponentially when new brands join the ecosystem.
  • Accessible documentation accelerates adoption more than formal training. Teams adopt what they can understand on their own.
  • A Design System is a product — it needs a roadmap, metrics, and continuous iteration. It's not a project with an end date.
